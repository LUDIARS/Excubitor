# Excubitor v0.1 設計書

> **ステータス**: ドラフト (実装着手前)
> **更新日**: 2026-05-12
> **対象バージョン**: v0.1

## 1. ミッション

LUDIARS 全サービス (28 リポ) の **可観測性 + 復旧操作 + 集中設定** を 1 つの UI で提供する中央サーバ。

- **死活監視**: docker / プロセスの running 状態 + health endpoint 結果
- **ログ集約**: 各サービスの stdout/stderr を中央へストリーム
- **エラータスク**: ログ中のエラーパターンを検知 → 人間に triage を促す内部 queue + UI
- **設定可視化**: 各サービスがどの compose / env / Infisical 設定で動いているかを表示
- **復旧操作**: ダウン中のサービスを UI から start / restart
- **Infisical 遠隔設定**: secret CRUD を Excubitor の UI から実行

Concordia から監視系コード (`/v1/services`, `/v1/processes`, `services-catalog.json`, `ProcessManager`,
`processes` / `process_logs` テーブル) を切り出し、Concordia は AI 人格・セッション協調専業に戻す。

**v0.1 はサーバ + 同居 agent の 1 ホスト構成のみ**。他ホスト agent 配備は v0.2 以降。

### 3.1 中央サーバ要件

- Linux + docker (or compatible runtime)
- infra Postgres (LUDIARS 共有) に到達可能
- 永続ストレージ (logs / error_tasks 保管)
- Tailscale 加入推奨 (agent からの WS 接続用、AWS/VPS agent 配備時)
- サーバ実機は v0.1 時点で未定 — 設計は「docker compose で 1 ホストに乗る」前提

### 3.2 配置単位

| ノード | 役割 |
|--------|------|
| **Excubitor Server** | API + UI + DB (infra) + log fan-out + error detector |
| **Excubitor Agent** | docker / process 情報を集めて server へ WS で push、control 命令を受けて docker compose 操作 |

v0.1 は同居 (server プロセスが docker socket を直接叩く)。v0.2 で独立プロセス化。

### 4.1 v0.1 (今回)

| 機能 | 詳細 |
|------|------|

## 5. データモデル

infra Postgres に新規 DB `excubitor` を追加 (`infra/init-databases.sql` + `infra/services.yaml` 更新)。

```sql
-- v0.1 minimum
services             -- catalog snapshot (yaml + DB の二重持ち、 yaml が source of truth)
hosts                -- agent 配置先 machine
service_instances    -- どの service が どの host で動いてるか (FK)
liveness_history     -- TTL 30 日、 service_instance ごとの probe 結果
process_logs         -- TTL 7 日、 source = service_instance、 level / line
error_tasks          -- triage queue
error_rules          -- regex / keyword + severity + service フィルタ
audit_log            -- operation 履歴
```

Concordia の `processes` / `process_logs` テーブルは Excubitor へ migration (schema は調整)。

## 6. API 概要

REST (Hono) + 必要に応じて WS / SSE。LUDIARS の Cernere 認証規約に準拠 (WS セッション or HTTP + JWT)。

```
GET    /api/v1/services                       — catalog 一覧 + liveness
GET    /api/v1/services/:id                   — 詳細 (config + last_probe)
POST   /api/v1/services/:id/control           — start | stop | restart
GET    /api/v1/services/:id/logs              — SSE log stream (since cursor)

GET    /api/v1/error-tasks                    — open / acknowledged / resolved
PATCH  /api/v1/error-tasks/:id                — triage (ack / resolve / dismiss / snooze)

GET    /api/v1/error-rules
POST   /api/v1/error-rules                    — regex / keyword 登録

GET    /api/v1/hosts                          — agent 一覧

GET    /api/v1/infisical/projects             — Infisical proxy (read-only)
GET    /api/v1/infisical/secrets              — list with redaction
POST   /api/v1/infisical/secrets              — create / update / delete
POST   /api/v1/infisical/bootstrap            — admin が credential を interactive 入力

GET    /api/v1/audit                          — 操作履歴

GET    /health
```

Agent ↔ Server (WS, v0.1 は server 内ローカルで完結、 v0.2 で外部化):

```json
// agent → server
{"type":"heartbeat","host":"...","ts":...,"docker":[{"name":"infra-postgres-1","state":"running",...}]}
{"type":"probe_result","service_instance_id":"...","ok":true,"latency_ms":12}
{"type":"log_line","service_instance_id":"...","ts":...,"level":"error","line":"..."}
{"type":"event","kind":"started|stopped|crashed","service_instance_id":"..."}

// server → agent
{"type":"control","service_instance_id":"...","action":"start|stop|restart"}
{"type":"probe_now","service_instance_id":"..."}
{"type":"set_interval","service_instance_id":"...","interval_sec":30}
```

## 7. Service catalog (YAML)

`catalog/services.yaml` が source of truth。例:

```yaml
services:
  - code: cernere
    name: Cernere
    repo: LUDIARS/Cernere
    runtime: docker-compose
    compose_file: E:/Document/Ars/Cernere/docker-compose.yaml
    services:                      # compose 内の service 名
      - backend-dev
      - frontend-dev
    health:
      type: http
      url: http://localhost:8080/health
      interval_sec: 30
    log_sources:
      - docker: cernere-backend-dev
      - docker: cernere-frontend-dev
    infisical:
      project_id: 19c9e77d-646c-4a0e-a091-b494234953ea
      environment: dev

  - code: memoria
    name: Memoria
    repo: LUDIARS/Memoria
    runtime: node
    cwd: E:/Document/Ars/Memoria
    command: npm run dev:server
    health:
      type: http
      url: http://localhost:3000/api/health
    log_sources:
      - file: logs/memoria.log
```

`runtime` は `docker-compose | docker | node | systemd (v0.3+) | windows-service (v0.3+)` を予定。
catalog の変更検知は server 起動時 + ファイル watch。

### 7.1 自動起動 (autostart)

Concordia の dev-process.md 自動起動機能を Excubitor で継承する。catalog のサービスに
`autostart: true` を付けると、Excubitor server 起動時に自動で spawn する。

```yaml
services:
  - code: memoria-server
    name: Memoria (server)
    runtime: node
    cwd: E:/Document/Ars/Memoria/server
    command: npm run dev
    autostart: true                # ← server 起動時に自動 spawn
    restart_policy: on-failure     # exit code != 0 で再起動
    max_restart: 5                 # 連続 5 失敗で停止 + error_task 起票
    log_sources:
      - process: stdout            # spawn 直結の stdout/stderr を log_stream へ
    health:
      type: http
      url: http://localhost:3000/api/health
      grace_period_sec: 10         # 起動直後の health 失敗は許容
```

**dev-process.md 互換モード**:
catalog から runtime: dev-process-md として参照すると、対象ディレクトリの `dev-process.md`
を読み取って (Concordia と同じフォーマットで) コマンドを抽出 → spawn。
既存 dev-process.md を持つリポはこの runtime で catalog 登録できる。

Concordia 側の SessionsRouter で行っていた「session 開始 hook → dev-process.md 自動起動」は廃止。
代わりに Excubitor 自身が catalog を見て常時自動起動を担う (= AI セッションと decoupled、
AI が動いていない時もサービスは立ち上がっている)。

### 7.2 Secret injection (`.env` 不要化)

**目的**: LUDIARS の各サービスは Infisical project_id / environment がほぼ共通で、それぞれの
リポに `.env` を持つのは管理が散らかる上に漏洩面積が広い。Excubitor から起動時に Infisical
secret を fetch して **子プロセス env として直接注入**、ファイルに一切残さないクリーンな
パイプを v0.1 から提供する。

#### 7.2.1 catalog 拡張

```yaml
services:
  - code: memoria-server
    runtime: node
    cwd: E:/Document/Ars/Memoria/server
    command: npm run dev
    autostart: true
    infisical:
      project_id: 19c9e77d-646c-4a0e-a091-b494234953ea
      environment: dev
      inject: true                  # ← Excubitor が fetch して env として注入
      prefix: ""                    # 任意。例: "MEMORIA_" を付けて namespace 化
      include: ["DATABASE_URL","REDIS_URL","*_KEY","*_SECRET"]  # glob 配列、 省略時は全件
      exclude: ["AWS_*","INTERNAL_*"]
```

Excubitor 側で:

1. spawn 直前に `(project_id, environment)` で Infisical から secret を fetch (Excubitor の
   bootstrap 済み credential を使用、§9.3)
2. include / exclude フィルタで対象を絞り込み
3. `process.env` + fetched secrets をマージして子プロセス env として渡す
4. **fetched secrets は process メモリのみで保持、ファイルへの書き込みなし**
5. 子プロセス終了で env が消えてメモリも回収

#### 7.2.2 runtime ごとの注入方式

| runtime | 実装 |
|---------|------|
| **node** / **dev-process-md** | `child_process.spawn(cmd, args, { env: { ...os, ...secrets } })` — 最もクリーン |
| **docker-compose** | docker compose の override file を **/dev/shm** (tmpfs) に書き出して `-f base.yml -f /dev/shm/excubitor-<uuid>.yml up -d`。子プロセス終了時に override file を unlink。tmpfs なので電源断時も残らない |
| **docker** (直接 run) | `docker run --env KEY=VALUE ...` をシェル引数として渡す。Excubitor の引数経由なので shell history / disk なし |

#### 7.2.3 既存 `.env` との互換戦略

各サービスのコード側で `dotenv` で `.env` を読む実装が残っていても問題ない:

- env が先に設定されている場合 dotenv は上書きしない (デフォルト挙動)
- `.env` ファイルがなければ dotenv は noop
- v0.1 では「サービスの `.env` を削除する」までは Excubitor の責務外。Excubitor は inject の
  パイプを提供するだけ。各リポでの `.env` 削除は別 PR (Memoria / Cernere / Custos etc.)

#### 7.2.4 セキュリティ

- secret は Excubitor process メモリと spawn された子プロセスの env のみに存在
- ファイル (`/dev/shm` 上の override 含む) は spawn 完了即 unlink、tmpfs なので RAM 上のみ
- ログ・stdout に secret 値が漏れないよう、Excubitor が捕捉した log_line に対して **値のマスク**
  を実施 (Infisical から取ってきた値の正規化リスト + redacter)
- audit_log には secret 値を残さない (key 名のみ + project/env id)

#### 7.2.5 制約と注意

- **中央サーバは Linux 前提** (`/dev/shm` を使用)。Windows サーバの場合は OS の一時ディレクトリに
  fallback するが、その場合は tmpfs ではないので「電源断時に残らない」保証が弱まる → v0.1 では
  Linux のみ正式サポート、Windows は best-effort
- Infisical bootstrap 前は inject 不能。catalog で `inject: true` のサービスを autostart に
  すると、bootstrap 待ち状態 (pending) になる → UI から bootstrap → 順次 spawn
- Infisical 側 secret 更新時の **hot reload は v0.1 ではしない** (再起動が必要)。v0.2 で
  Infisical webhook or polling で差分検知 → サービス再起動の自動化を検討

## 9. 認証・セキュリティ

### 9.1 UI ↔ Server

Cernere user session 必須。Excubitor は Cernere の `excubitor` project として登録、Cernere の org 単位で
ACL を引く (admin / member)。WS 接続は session token 検証。

### 9.2 Server ↔ Agent (v0.2)

- mTLS (証明書ベース、 server が CA、 agent ごとに証明書発行)
- もしくは Cernere PeerAdapter (B2B service-key)
- v0.1 は同居なのでネットワーク不要

### 9.3 Infisical credential の扱い

**ユーザ要望**: 「起動時に Infisical に問い合わせる。OS にも .env にもログは残さない」

設計:

1. Excubitor server 起動直後、Infisical 機能は **未初期化** 状態
2. 管理者が UI / API (`POST /api/v1/infisical/bootstrap`) から
   `INFISICAL_CLIENT_ID` + `INFISICAL_CLIENT_SECRET` (universal-auth) を入力
3. Excubitor server は受け取った credential で Infisical universal-auth → access_token を取得
4. **access_token と元 credential は process メモリ内のみ保持**
   - 永続化なし (DB / .env / OS keyring 一切なし)
   - access_token の expiry に合わせて自動 refresh (credential はメモリにある間)
5. Excubitor server 再起動 = メモリ消失 = 管理者が再 bootstrap する運用

**影響**:
- 再起動コストが高い → 中央サーバの可用性を意識
- 自動運用 (cron 等) で Infisical secret を触る必要がある場合は、その操作だけ Infisical agent
  (Infisical 公式の Operator) を別途立てる選択肢を残す (v0.3+)
- bootstrap 操作自体は audit_log に記録

### 9.4 Audit

すべての mutation (control / triage / secret CRUD / bootstrap) を `audit_log` に記録。
Cernere user identity + IP + ts + payload (redacted) を保存。
